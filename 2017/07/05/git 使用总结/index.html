git 使用总结

两个主分支：

1）master ：只能用来包括产品代码，不能直接工作在这个分支，也不能直接将改动后的代码提交到这个分支上；

2）develop：进行任何新的开发的基础分支，当开始一个新的功能分支时，它讲是开发的基础。另外，该分支也汇集所有已完成的功能，并等待被整合到master分支中。

三个辅助分支：

1）feature(基于develop)：为即将发布的版本开发新功能特性，完成后merge回develop；

2）release(基于develop)：准备要发布版本的分支，用来修复bug，完成后merge回develop和master；

3）hotfix(基于master)：修复master上的问题，等不及release版本就必须马上上线，基于master，完成后merge回master和develop。

命令

  1. git flow init
    初始化

  2. git tag -a v0.0.1 -m “标记v1.0版本”
    给master打上tag

//=====================================================
// feature:为即将发布的版本开发新功能，基于develop
//=====================================================

  1. git flow feature start add_feature
    开发新特性，这个分支基于develop，创建add_feature分支，并切换到这个分支下。后面的add_feature自己取，但是最好取得有意义，比如你要完成地图功能map_feature

  2. git flow feature finish add_feature
    完成开发新特性，会自动合并add_feature分支到develop，然后删除add_feature分支，最后切回到develop分支

  3. git flow feature publish add_feature
    如果是团队开发，你想把新特性分支发布到远程服务器让其他组员使用，所以使用这命令

  4. git flow feature pull add_feature
    取得其他用户发布的新特性分支

//=====================================================
// release:准备要发布的分支，用来修复bug，基于develop
//=====================================================

  1. git flow release start v0.1.0
    从develop分支创建一个release分支

  2. git flow release publish v0.1.0
    提交修改后的代码，供其他组员使用

  3. git flow release track v0.1.0
    签出release版本的远程变更

  4. git flow release finish v0.1.0
    别忘了git push –tags
    完成release版本，自动执行了：(1)归并release分支到master；(2)用release分支名打tag；(3)归并release分支到develop；(4)移除release分支

    hotfix:紧急修复bug的分支，基于master

  5. git flow hotfix start v0.0.3
    创建紧急修复的分支

  6. git flow hotfix finish v0.0.3
    完成了紧急修复，代码归并回develop和master分支，相应的，mater分支打上修正版本的tag

git commit 规则

  1. git cz

feat：新功能（feature）
fix：修补bug
docs：文档（documentation）
style： 格式（不影响代码运行的变动）
refactor：重构（即不是新增功能，也不是修改bug的代码变动）
test：增加测试
chore：构建过程或辅助工具的变动

1、第一次运行git，必须先配置用户信息。
比如：
git config –global user.name “chenjiangchuan”
git config –global user.email jiangchuanc@gmail.com

2、查看配置信息
git config –list

3、从服务器的仓库克隆代码
git clone https://git.oschina.net/chenjiangchuan/hellobaby.git

4、检查当前当前文件状态
git status

5、查看当前所在的分支
git branch

6、创建分支
git branch 分支名

7、切换分支
git checkout 分支名

8、合并分支
git merge 分支名

9、删除分支
git branch -d 分支名

10、跟踪新文件
git add 文件名

11、跟踪完以后，就提交跟新
git commit -m “备注”

12、删除文件
git rm 文件名

13、查看提交历史
git log
git reflog 打印的信息更简洁

14、推送数据到远程仓库
git push origin 当前的分支名

15、本地分支和远程仓库不一样sha
git fetch origin
git merge origin/当前的分支名

16、重置到当前的版本
git reset — hard HEAD

17、回到上一个版本
git reset — hard HEAD^

18、回到某一个版本
git reset –hard 版本号 (使用git reflog查看版本号)

19、删除远程仓库的某一个分支
以 删除develop为例
git push origin :develop

团队开发

1、新建本地仓库
git init —bare

2、git clone 本地仓库 存放库的文件夹

3、git push 把本地代码提交到本地仓库

4、git pull 从本地仓更新最新代码

5、解决冲突：
（1）先从仓库更新代码：git pull
（2）解决冲突
（3）git add
（4）git commit -m
（5）提交版本：git push

tag的使用

1）创建标签
git tag -a V1.2 -m ‘WebSite version 1.2’

2）查看标签
git tag

3）git show V1.2

4）tag 远程推送
git push origin –tags

5）删除本地标签
git tag -d V1.2

6）删除仓库标签
push origin :refs/tags/V1.2

7）如何获取远程版本
git fetch origin tag V1.2

1）标记版本v1.0
git tag -a v1.0 -m “标记v1.0版本”

2）push到远程共享库
git push origin v1.0

3）v1.0出现bug了，通过标签创建一个分支用来修复bug
git checkout v1.0 -b v1.0fixBug

4）合并分支
git merge origin/v1.0fixbug -m “合并分支”

5）git push

1、error: The following untracked working tree files would be overwritten by merge:
git clean -d -fx
一个项目想指向不同的远程仓库

1、 git remote add upstream git@github.com:drubear/echarts.git
其中upstream是本地指向远程的名字，默认有一个origin指向另一个远程仓，所以这里名字
要不一样；

2、提交代码
git push upstream master

upstream是本地对应远程仓库的映射
master是要上传的分支

3、当发生repo主机ip发生变化，repo迁移问题，需要修改origin中指向的地址：
先使用git remote -v参看origin指向的url地址；

4、git remote set-url origin <新的git repo地址>
比如：git remote set-url origin git@github.com:google/gson.git
